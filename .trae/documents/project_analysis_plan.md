# 项目分析与后续开发规划 (Project Analysis & Development Plan)

## 1. 摘要 (Summary)
本项目 (`obsidian-project`) 是一个功能丰富、架构严谨的 Obsidian 任务与项目管理插件（类 Jira 体验）。当前处于成熟且活跃的开发期（v1.1.1），测试覆盖率高，功能完备。目前的重点正逐渐从新功能开发转向系统加固与性能优化。

## 2. 当前状态分析 (Current State Analysis)
- **架构设计**：采用高度模块化的清晰架构（如 `domain`, `index`, `markdown`, `repositories`, `services`, `views` 等），符合生产级标准。
- **核心能力**：基于本地 Markdown 的任务存储、自定义项目代码（如 PROJ-1）、多维度视图（看板、仪表盘）、强大的数据迁移与一致性校验。
- **代码质量**：拥有极高的测试覆盖率（涵盖单元测试到启动性能测试），严格遵守安全性与隐私规范（本地优先，无隐藏遥测）。

## 3. 当前风险及优先级 (Prioritized Risks)
根据对代码库的探索，当前项目存在以下技术债与风险（按优先级从高到低排序）：

### 🔴 高优先级风险：核心类臃肿 (God Object)
- **问题描述**：`src/services/project-manager.ts` 文件体积过大（超过 1100 行），集成了标签管理、人员管理、配置管理和迁移等多重逻辑。
- **影响**：违反了项目中定义的“单文件不超过 200-300 行”的规范，极大增加了后续维护、测试和多人协作的成本，容易引入回归 Bug。

### 🟠 中优先级风险：启动性能与内存压力 (Performance & Memory)
- **问题描述**：插件启动时需要扫描并解析 Vault 中的大量任务文件构建内存索引。虽然使用了并发池 (`mapConcurrent`)，且有相关性能测试，但在包含数千个任务的大型 Vault 中仍可能导致启动延迟。
- **影响**：全量元数据常驻内存对移动端或低内存设备极不友好，可能导致应用卡顿或崩溃。

### 🟡 低优先级风险：解析器健壮性与路径耦合 (Robustness & Coupling)
- **问题描述**：
  1. 深度依赖自定义的正则表达式解析 Markdown 提取元数据。面对用户极具个性的排版时，存在边缘解析失败的风险。
  2. 系统逻辑对特定目录结构（如默认的 `项目管理/全局配置.md`）存在一定耦合，对用户的 Vault 组织方式有一定要求。
- **影响**：可能导致部分任务无法正确被索引，或在用户重构目录时出现路径失效问题。

## 4. 后续建议开发方向 (Proposed Development Directions)

基于上述风险，建议后续的开发工作按照以下方向分阶段进行：

### 阶段一：架构重构与解耦 (解决高优先级风险)
- **重构 `ProjectManager`**：将 `src/services/project-manager.ts` 拆分为更小、职责更单一的服务。
  - 提取 `TagManagerService` 处理标签逻辑。
  - 提取 `PersonnelService` 处理人员管理逻辑。
  - 提取 `MigrationService` 专注于数据迁移。
- **预期收益**：提升代码可读性与可维护性，降低核心模块的复杂度。

### 阶段二：性能与内存优化 (解决中优先级风险)
- **引入懒加载与增量索引**：
  - 优化启动流程，仅在启动时加载必要的核心数据，推迟详细任务内容的解析。
  - 监听 Obsidian 的 `modify`, `create`, `delete` 事件，维护增量索引，避免全量重构。
- **移动端适配优化**：针对 `isDesktopOnly` 为 `false` 的场景，增加内存占用限制或分页加载策略。

### 阶段三：增强健壮性与用户体验 (解决低优先级风险)
- **强化 Markdown 解析器**：引入更健壮的 AST（抽象语法树）解析方案（如基于 `remark` 体系）替代部分脆弱的正则匹配，提升解析容错率。
- **灵活的路径配置**：减少对特定层级目录的硬编码依赖，允许用户在设置中完全自定义存储结构，并提供目录变更时的自动修正工具。

## 5. 验证与后续步骤 (Verification & Next Steps)
- **确认计划**：请用户审查本规划，确认优先级是否符合业务预期。
- **开始执行**：一旦确认，将优先创建 `ProjectManager` 的重构方案并逐步实施，确保所有现存测试用例（尤其是集成测试）在重构后依然通过。
