# 任务输入实时 Markdown 预览评估

## 结论

可实现，但不建议直接把当前 `TextAreaComponent` 替换成完整所见即所得编辑器。推荐先实现“编辑 / 预览”分栏或标签页，再评估嵌入 CodeMirror 6 的实时 Markdown 编辑体验。

## 可选方案

### 方案 A：输入区与实时预览区并排

- 左侧保留原生 textarea，右侧使用 Obsidian `MarkdownRenderer.render`。
- 输入事件做 150–250ms 防抖，避免每次按键完整重绘。
- 优点是实现边界清晰、兼容现有正文和备注字符串、移动端可降级为编辑/预览标签页。
- 风险是包含嵌入、查询或复杂插件语法时，预览成本较高；必须正确管理渲染组件的 unload 生命周期。

### 方案 B：CodeMirror 6 编辑器加预览

- 使用 Obsidian 自带编辑器依赖或独立 CodeMirror 状态创建 Markdown 编辑器。
- 可以提供语法高亮、自动缩进、列表续写和快捷键。
- 风险是 Obsidian 未公开保证内部编辑器扩展 API 稳定，维护成本和移动端输入法兼容风险更高。

### 方案 C：完全所见即所得

- 需要处理光标映射、Markdown 源码往返、嵌套列表、链接、嵌入和第三方语法。
- 对当前插件收益不足，不建议现阶段实施。

## 推荐实施顺序

1. 为正文和备注增加“编辑 / 预览”模式，桌面端允许左右分栏，移动端使用标签页。
2. 使用 `MarkdownRenderer`、200ms 防抖和独立 `Component` 管理资源。
3. 记录大文本和嵌入内容的渲染耗时，超过阈值时仅在用户切换到预览后渲染。
4. 用户验证通过后，再评估是否需要 CodeMirror 6 语法高亮。

本轮仅完成评估，没有修改新增或编辑任务的输入行为。
